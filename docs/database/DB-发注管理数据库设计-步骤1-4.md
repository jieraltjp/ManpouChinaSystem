# 发注管理数据库设计 — 步骤 1~4

> **版本**: 1.0.0
> **更新**: 2026-04-21
> **依据**: `DOMAIN-发注管理领域模型.md` v1.3.0
> **数据库**: MySQL 8.x / InnoDB / `utf8mb4_unicode_ci`

---

## §1 表清单

| # | 表名 | 用途 | 状态 |
|---|------|------|------|
| 1 | `replenishment_demand` | 补货需求单（第一步） | **新增** |
| 2 | `factory` | 工厂信息（第二步） | **新增** |
| 3 | `procurement` | 发注单（第二步，已存在，**扩展字段**） | 修改 |
| 4 | `product` | 商品目录（所有步骤，已存在骨架，**扩展字段**） | 修改 |
| 5 | `qc_record` | 验货记录（第三步） | **新增**（替代原内嵌 QcRecord 值对象） |
| 6 | `logistics_plan` | 调配计划（第四步） | **新增** |
| 7 | `container` | 货柜信息（第四步） | **新增** |
| 8 | `consolidation_pool` | 拼柜池（第四步） | **新增** |
| 9 | `finance_record` | 财务结算（与现有 FinanceRecord 对齐） | **新增** |
| 10 | `return_record` | 退货记录（通用） | **新增** |

> 表 3（procurement）和表 4（product）**仅扩展字段，不改变现有列**。

## §2 命名规范

| 层级 | 规则 | 示例 |
|------|------|------|
| 表名 | `snake_case`，模块前缀 | `replenishment_demand` |
| 列名 | `snake_case`，全小写 | `demand_type` |
| 主键 | `id`（单表） | `id BIGINT PK` |
| 外键 | `{referenced_table_singular}_id` | `factory_id` |
| 枚举列 | ENUM 类型 | `status ENUM(...)` |
| 审计字段 | 继承 `BaseEntity` | `create_by / create_time / update_by / update_time / is_deleted` |

## §3 审计字段（BaseEntity）

所有业务表必须继承以下审计字段（逻辑删除模式）：

| 列名 | MySQL 类型 | Java 字段 | 说明 |
|------|-----------|-----------|------|
| `id` | `BIGINT PK AUTO_INCREMENT` | `Long id` | 主键 |
| `create_by` | `VARCHAR(64)` | `String createBy` | 创建人 |
| `create_time` | `DATETIME(6)` | `LocalDateTime createTime` | 创建时间 |
| `update_by` | `VARCHAR(64)` | `String updateBy` | 更新人 |
| `update_time` | `DATETIME(6)` | `LocalDateTime updateTime` | 更新时间 |
| `is_deleted` | `BIT(1)` | `Boolean isDeleted` | 逻辑删除（0=正常, 1=删除） |

## §7 开发注意事项

1. **Hibernate ddl-auto**：`ddl-auto: update` 会自动同步表结构，新增列不会丢失数据。
2. **product 表名冲突**：现有 `product_example` 骨架表需先重命名或删除，再创建真实 `product` 表。
3. **枚举变更**：MySQL `ENUM` 列增加值后，需同步更新 Java enum。
4. **JSON 字段**：`images` 列使用 JSON 数组格式存储图片 URL 列表，Java 端用 `ObjectMapper` 或 Jackson 处理。
5. **流水号格式**：
   - `replenishment_demand.demand_code` → `DM-YYYYMMDD-NNN`
   - `factory.factory_code` → `F-YYYYMMDD-NNN`
   - `qc_record.qc_code` → `Q-YYYYMMDD-NNN`
   - `logistics_plan.plan_code` → `L-YYYYMMDD-NNN`
   - `consolidation_pool.pool_code` → `P-YYYYMMDD-NNN`
   - `finance_record.finance_code` → `FN-YYYYMMDD-NNN`
   - `return_record.return_code` → `RT-YYYYMMDD-NNN`
