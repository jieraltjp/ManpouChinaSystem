# 页面规格 — 订单总览（核心视图）

> **版本**: 1.0.0
> **创建**: 2026-04-22
> **路由**: `/procurement/overview`
> **组件**: `OrderOverviewPage.vue`
> **对应后端**: `GET /api/v1/orders/{id}/overview` 聚合接口
> **依赖文档**: `SPEC-B01-全链路总览.md` · `SPEC-B09-订单总览-API设计.md`
> **前置**: 至少有一条 Procurement 记录

---

## 1. 页面定位

**订单总览是整个系统的核心视图**。以一条 Procurement（发注单）为锚点，横向聚合该订单从「需求录入」到「运营销售」的全链路信息。解决业务痛点：目前各步骤数据分散在独立页面，需要来回切换才能看到完整订单生命周期的现状。

**设计原则**：
- **锚点驱动**：以 Procurement.id 为唯一入口
- **单向信息流**：上一步信息流入下一步，下一步不可修改上游数据
- **状态可见**：8步状态一览，当前卡点一目了然
- **可操作**：每一步可直接跳转编辑（权限范围内）

## 2. 入口与路由

### 2.1 入口

| 入口位置 | 触发行为 |
|---------|---------|
| 发注单列表操作列 `[总览]` | 进入 `/procurement/overview/{procurementId}` |
| 验货列表操作列 `[总览]` | 进入 `/procurement/overview/{procurementId}` |
| 调配列表操作列 `[总览]` | 进入 `/procurement/overview/{procurementId}` |
| 侧边栏菜单 | 进入 `/procurement/overview`（打开采购单选择器） |

### 2.2 路由

```
/procurement/overview                  → 采购单选择器（无ID时）
/procurement/overview/:procurementId  → 订单总览详情页
```

## 4. 步骤卡片字段总览

### 步骤1 — 补货需求（ReplenishmentDemand）

| 字段 | 说明 |
|------|------|
| `demandCode` | 需求编号 |
| `demandType` | 补货 / 新品采购 |
| `productCode` + `subProductCode` | 货号 |
| `quantity` | 需求量 |
| `destination` | 目的地 |
| `japanLead` | 日本担当 |
| `status` | PENDING / CONVERTED / CANCELLED |

**操作**：`[编辑]`（PENDING 时）

---

### 步骤2 — 发注单（Procurement）

## 5. 状态进度条

横穿页面顶部的 8 步进度条，每步用图标+颜色表示状态：

| 步 | 名称 | 完成 | 进行中 | 未开始 |
|----|------|------|--------|--------|
| 1 | 补货 | ● 蓝色 | ◐ 蓝色 | ○ 灰色 |
| 2 | 发注 | ● 蓝色 | ◐ 橙色 | ○ 灰色 |
| 3 | 验货 | ● 绿色 | ◐ 蓝色 | ○ 灰色 |
| 4 | 调配 | ● 蓝色 | ◐ 橙色 | ○ 灰色 |
| 5 | 国报 | ● 蓝色 | ◐ 蓝色 | ○ 灰色 |
| 6 | 日报 | ● 蓝色 | ◐ 蓝色 | ○ 灰色 |
| 7 | 退税 | ● 绿色 | ◐ 蓝色 | ○ 灰色 |
| 8 | 运营 | ● 绿色 | ◐ 橙色 | ○ 灰色 |

- **完成**（●）：该步骤对应记录已创建且状态为终态或进行中
- **进行中**（◐）：有记录，状态非终态
- **未开始**（○）：无对应记录
