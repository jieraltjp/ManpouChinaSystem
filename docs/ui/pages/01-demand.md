## 1. 页面定位

补货需求的入口页面。**每条需求单 = 一个子货号**（商品唯一标识 = 主货号 + 子货号）。
非新品走补货流程，新品走新品采购流程。录入后转为发注单（Procurement），进入步骤2。

---

## 2. 核心设计原则

- **商品唯一标识 = 主货号 + 子货号**，如 `ad009-be`
- **每条 ReplenishmentDemand = 一个子货号**（不在 JSON 数组里存多个子货号）
- 转采购：**1 Demand → 1 Procurement**（一对一）
- **表单分两种录入模式**：
  - **单个录入**：一次创建 1 条需求（1 主货号 × 1 子货号 × 1 目的地）
  - **批量录入**：一次创建 N 条需求（1 主货号 × N 个子货号，各有独立数量和目的地）

│  目的地   [___________久留米___________]                      │
│                                                              │

**字段规范**：

| 字段 | 控件 | 必填 | 说明 |

| 主货号 | `el-select`（搜索远程） | ✅ | 选择后自动加载子货号列表 |
| 子货号 | `el-select`（下拉） | ✅ | 等主货号选择后启用，显示「颜色名 (子货号)」 |
| 需求量 | `el-input-number` | ✅ | min=1，controls 在右侧 |
| 目的地 | `el-input` | | 文本输入 |
| 日本担当 | `el-input` | | 文本输入 |
| 备注 | `el-input` | | 文本输入，跨满行 |

**布局规范**：

- 子货号、需求量、目的地三字段从左到右并排：`flex:2 1 200px` `flex:1 1 120px` `flex:2 1`
- 日本担当和备注在最后一行，备注跨满宽度

### 5.4 批量录入模式

适用场景：同一主货号下多个子货号，各有不同的数量或目的地，一次创建 N 条需求单。

```
┌─────────────────────────────────────────────────────────────┐
│ 【基本信息】                                                  │
│  需求类型  [● 补货  ○ 新品采购]                               │
│  主货号   [___________搜索/选择主货号___________▼]            │
│  日本担当 [___________田中___________]                         │
│                                                              │
│ 【需求明细】← 批量模式，此区域为列表形式                       │
│  ┌─────────────────┬──────────┬─────────────────┬─────┐    │
│  │ 子货号（颜色）   │ 需求量    │ 目的地           │ 操作 │    │
│  ├─────────────────┼──────────┼─────────────────┼─────┤    │
│  │ [ad009-be ▼]   │ [___100] │ [____久留米___] │ [×] │    │
│  │ [ad009-bu ▼]   │ [___ 50] │ [____名古屋___] │ [×] │    │
│  └─────────────────┴──────────┴─────────────────┴─────┘    │
│                                           [+ 添加一行]       │
│                                                              │
│  共 2 条子货号                                               │
│                                              [取消]  [保存]  │
└─────────────────────────────────────────────────────────────┘
```

**字段规范**：

| 区域 | 字段 | 控件 | 说明 |
|------|------|------|------|
| 基本信息 | 需求类型 | `el-radio-group` | 同单个模式 |
| 基本信息 | 主货号 | `el-select` | 选择后自动加载子货号下拉选项 |
| 基本信息 | 日本担当 | `el-input` | 整批共用 |
| 需求明细 | 子货号 | `el-select`（下拉，可搜索） | 等主货号选择后启用 |
| 需求明细 | 需求量 | `el-input-number` | min=1 |
| 需求明细 | 目的地 | `el-input` | 文本输入 |
| 需求明细 | 操作 | 按钮 | [×] 删除当前行 |

**交互规范**：
- `[+ 添加一行]` 按钮：追加一行空白子货号记录，默认代入主货号第一个子货号
- 每行子货号下拉：启用过滤，已选过的子货号在全局列表中灰显（禁止重复选择）
- 表单提交前校验：至少 1 条子货号明细；子货号不能重复；每条数量 ≥ 1
- 保存时：循环调用 `POST /api/v1/demands`，全部成功才算完成

### 5.5 业务规则

- 选择主货号后，自动加载该主货号下所有子货号（颜色变体）
- 单个录入：用户手动选择子货号（一个）
- 批量录入：同一主货号 × N 个子货号，各自有独立数量/目的地，每条独立创建 Demand

### 5.6 提交

**单个模式**：

```
POST /api/v1/demands
{
  "demandType": "REPLENISHMENT",
  "productCode": "ad009",
  "subProductCode": "ad009-be",
  "quantity": 100,
  "destination": "久留米",
  "japanLead": "田中",
  "remarks": ""
}
```

**批量模式**（循环 N 次 POST）：

```
POST /api/v1/demands  × N（每条子货号一行）
```

成功后刷新列表；弹窗关闭。

## 6. 转采购（v2.0.0 — 1:1 模式）

### 6.1 触发条件

状态 = `PENDING`，操作列点击 `[转采购]`。

### 6.2 行为

1. 弹出工厂选择弹窗
2. 选择工厂后确认
3. **1 Demand → 1 Procurement**（该子货号独立生成一条 Procurement）
4. 需求单状态 → CONVERTED
5. `linkedProcurementId` 记录对应的 Procurement.id

### 6.3 API 调用

```
POST /api/v1/demands/{id}/convert
请求体：{ "factoryId": 123 }
响应：{ "code": "ok", "data": { "demandStatus": "CONVERTED", "linkedProcurementId": 101 } }
```

### 6.4 查看采购单（CONVERTED）

操作列 `[查看采购单]` → 弹窗显示对应的 Procurement 明细

### 6.5 撤销转换

操作列 `[撤销转换]` → 删除关联的 Procurement，回滚需求状态 → PENDING

## 8. API 集成
