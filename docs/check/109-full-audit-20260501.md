## 二、后端审计（manpou-allinone）

### 2.1 Domain Model 层 — 整体良好

**做得好的：**
- `BaseEntity` 统一 `id/createTime/updateTime/createBy/updateBy/deleted`
- 领域逻辑在实体中（如 `Procurement.calculateEstimatedPriceJpy()`）
- 无 JPA 关联，所有关联用外键 `Long xxxId`，减少耦合
- 跨模块通过 Port 接口解耦（`ProcurementPort`/`QcQueryPort`）

**问题：**
| # | 问题 | 文件 | 影响 |
|---|------|------|------|
| 1 | `Product.java`（197行）承载 30+ 字段（报关/尺寸/仓库） | `product/domain/model/Product.java` | 职责不清，建议拆分 |
| 2 | `ProductCategory` 枚举在 `procurement` 和 `product` 两处重复定义 | 两个 domain 模块 | 代码重复，应统一到 `common` |

### 2.2 UseCase 层 — 136 处 @Transactional，优秀

**严重反模式：内存分页（3处 P0）**

| 文件 | 方法 | 问题 |
|------|------|------|
| `SalesRecordUseCase.java:30` | `pageQuery()` | `findAll()` → 全表加载到 JVM 内存 → `stream().filter()` |
| `SalesRecordUseCase.java:126` | `getAlerts()` | 同上，全表加载后内存过滤 |
| `JapanCustomsUseCase.java:33` | `pageQuery()` | 同上 |

**修复方向：** 将过滤条件下沉到 Repository 的 `Page<...> findByXxx(...)` 方法，用数据库层分页。

**其他 UseCase 问题：**

| # | 问题 | 文件 | 说明 |
|---|------|------|------|
| 1 | `entity.setXxx()` 后重复调用 assembler（第 87 行无用调用） | `ProcurementUseCase.java:87` | 应删除 |
| 2 | self-invocation 绕过事务代理 | `JapanCustomsUseCase.java:68` | `batchCreate()` → `create()`，`@Transactional` 被绕过，但整体有事务所以当前可工作 |

### 2.3 Repository 层 — N+1 风险 + 缺失索引

**N+1 查询（2处 P1）：**

```java
// ProductUseCase.getProductFactories() — 循环中逐条查
links.stream().map(link -> {
    factoryRepository.findByIdAndDeletedIsFalse(link.getFactoryId()) // 每条记录一次查询
}).toList();
// 应改为 findAllByIdInAndDeletedIsFalse(factoryIds) 批量查
```

**缺失索引（P1）：** Entity 定义了 `@Index`，但 Flyway 脚本中无 `CREATE INDEX` 语句：

| Entity | 缺失索引数 |
|--------|-----------|
| `Procurement` | 5 个 |
| `ReplenishmentDemand` | 7 个 |
| `Product` | 5 个 |
| 合计 | **17+ 个索引** |

当前 Hibernate `ddl-auto: update` 自动建索引（开发环境），但生产启用 Flyway 后索引不会自动创建。

### 2.4 事件监听器 — 4 处缺少 @Transactional（P1）

以下监听器使用 `phase = AFTER_COMMIT`，原始事务已提交，监听器方法运行在**无事务上下文**中：

| 监听器 | 方法 | 写操作 |
|--------|------|--------|
| `LogisticsInTransitEventListener` | `createDomesticCustomsRecord()` | `domesticCustomsRepository.save()` |
| `NewPurchaseDemandEventListener` | `createNewPurchaseDemand()` | `demandRepository.save()` |
| `SalesLowStockEventListener` | `createReplenishmentDemand()` | `demandRepository.save()` |
| `JapanCustomsClearedEventListener` | `createSalesRecordFromProcurement()` | `salesRecordRepository.save()` |
| `ProcurementQcPassedEventListener` | `advanceProcurementStatus()` | `procurementRepository.save()`（异常被静默吞掉 P2） |

**修复：** 为这些 public 方法添加 `@Transactional(propagation = Propagation.REQUIRES_NEW)`。

### 2.6 内网序列号回滚风险（P2）

```java
// LogisticsInTransitEventListener.java:31
private static final AtomicLong SEQ = new AtomicLong(System.currentTimeMillis() % 1000);
```
进程重启后序列号回滚，同一日期内可能生成重复 `customsCode`（如 `DC-20260501-001`），且无数据库唯一约束保护。
