## 三、问题详项

### 🔴 P0 — 字段映射不一致：BaseEntity.deleted vs is_deleted

**现象**: `BaseEntity` 实体类定义的字段名为 `deleted`（Boolean），但所有表列名均为 `is_deleted`。

```java
// BaseEntity.java（实体）
@Column(name = "is_deleted", nullable = false)
private Boolean deleted = false;  // ← 字段名 deleted，列名 is_deleted
```

**影响**:
- 代码可读性：字段名 `deleted` 与 DB 列名 `is_deleted` 不一致，阅读代码时易混淆
- Lombok `@Getter` 生成 `isDeleted()` 方法（因字段名是 `deleted`，生成的是 `getDeleted()` 或 `isDeleted()` 取决于 Lombok 版本）
- 所有子实体均通过 `Boolean.TRUE.equals(this.deleted)` 判断删除状态

**建议**: 重命名 `BaseEntity.deleted` → `BaseEntity.isDeleted`，同时更新所有子实体引用点（预计影响较小，属底层修改）。

**影响范围**: 所有 15 张业务表（procurement / factory / product / product_factory / replenishment_demand / qc_record / logistics_plan / domestic_customs_record / japan_customs_record / tax_refund_record / sales_record 等）

---

### 🔴 P0 — 缺失软删除索引（is_deleted 列无索引）

所有业务表均在 `is_deleted` 列上有大量查询（`WHERE is_deleted = FALSE`），但以下表缺少 `idx_xxx_is_deleted` 索引：

| 表名 | 文档要求索引 | 实体中是否声明 | 状态 |
|------|------------|--------------|------|
| `product` | `idx_product_is_deleted` | ❌ 未声明 | 🔴 缺失 |
| `sales_record` | `idx_sr_is_deleted` | ❌ 未声明 | 🔴 缺失 |
| `logistics_plan` | `idx_lp_is_deleted` | ❌ 未声明 | 🔴 缺失 |
| `domestic_customs_record` | `idx_dc_is_deleted` | ❌ 未声明 | 🔴 缺失 |
| `japan_customs_record` | `idx_jp_is_deleted` | ❌ 未声明 | 🔴 缺失 |
| `tax_refund_record` | `idx_tr_is_deleted` | ❌ 未声明 | 🔴 缺失 |
| `procurement` | `idx_proc_is_deleted` | ❌ 未声明 | 🔴 缺失 |

**说明**: `replenishment_demand` / `qc_record` / `factory` 的 `is_deleted` 索引已正确声明。

**修复 SQL**:
```sql
-- product
ALTER TABLE product ADD INDEX idx_product_is_deleted (is_deleted);
-- sales_record
ALTER TABLE sales_record ADD INDEX idx_sr_is_deleted (is_deleted);
-- logistics_plan
ALTER TABLE logistics_plan ADD INDEX idx_lp_is_deleted (is_deleted);
-- domestic_customs_record
ALTER TABLE domestic_customs_record ADD INDEX idx_dc_is_deleted (is_deleted);
-- japan_customs_record
ALTER TABLE japan_customs_record ADD INDEX idx_jp_is_deleted (is_deleted);
-- tax_refund_record
ALTER TABLE tax_refund_record ADD INDEX idx_tr_is_deleted (is_deleted);
-- procurement
ALTER TABLE procurement ADD INDEX idx_proc_is_deleted (is_deleted);
```

### 🟡 P1 — 索引/UK 声明不一致

#### P1-1: `factory.factory_name` — UK vs INDEX

| 位置 | 声明 |
|------|------|
| DB-10 建表语句 | `UNIQUE KEY uk_factory_name (factory_name)` |
| DB-10 ALTER 语句 | 未重提 UK |
| Java 实体 `Factory.java` | `@Index(name = "idx_factory_name", columnList = "factory_name")` |

**问题**: DB 设计要求 `factory_name` 为唯一约束，但实体只声明了普通索引（INDEX）。

**影响**: Hibernate ddl-auto 可能只创建 INDEX 而非 UNIQUE KEY，导致 `factory_name` 允许重复。

**修复**: 实体中 UK 声明为：
```java
@Table(name = "factory",
    uniqueConstraints = @UniqueConstraint(name = "uk_factory_name", columnNames = "factory_name"),
    indexes = { ... }
)
```

#### P1-2: `japan_customs_record.customs_entry_no` — UK vs INDEX

| 位置 | 声明 |
|------|------|
| DB-06 建表语句 | `UNIQUE KEY idx_jp_customs_entry_no (customs_entry_no)` |
| Java 实体 `JapanCustomsRecord.java` | 仅 `@Index(name = "idx_jp_procurement_id", ...)` |

**问题**: `customs_entry_no` 应为 UK，实体未声明。

#### P1-3: `logistics_plan.plan_code` — INDEX vs UK

| 位置 | 声明 |
|------|------|
| DB-04 建表语句 | `UNIQUE KEY uk_plan_code (plan_code)` |
| Java 实体 `LogisticsPlan.java` | `@Index(name = "uk_plan_code", columnList = "plan_code", unique = true)` |

**说明**: 实体声明了 `unique = true`，Hibernate 实际生成 UNIQUE INDEX，与 DB 设计一致。✅ 此项一致。

#### P1-4: `qc_record.qc_code` — UK 一致

| 位置 | 声明 |
|------|------|
| DB-03 建表语句 | `UNIQUE KEY uk_qc_code (qc_code)` |
| Java 实体 `QcRecord.java` | `@UniqueConstraint(name = "uk_qc_code", columnNames = "qc_code")` |

✅ 一致。

#### P1-5: `domestic_customs_record` — 缺少多处索引

| 索引名 | DB-05 要求 | 实体声明 |
|--------|-----------|---------|
| `idx_dc_procurement_id` | ✅ | ❌ 未声明 |
| `idx_dc_logistics_plan_id` | ✅ | ❌ 未声明 |
| `idx_dc_status` | ✅ | ❌ 未声明 |
| `idx_dc_is_deleted` | ✅ | ❌ 未声明 |

**说明**: `DomesticCustomsRecord` 实体完全未声明任何索引（`@Table(name = "domestic_customs_record", ...)` 只有一个 UK），全部依赖 Hibernate ddl-auto 自动推断。

### 🟢 P2 — 字段顺序优化建议

#### P2-1: `product` 表字段顺序调整

**现状**（Entity 声明顺序）:
```
id → master_code → sub_code → jan_code → name_ja → name_en → name_zh → image_url
→ color_name → material → category → status → origin → unit → quantities → carton_qty
→ amount_rmb → material_ja → length_cm → width_cm → height_cm → volume_cbm
→ net_weight_kg → gross_weight_kg → unit_price_rmb → tax_point → tax_rate
→ hs_code → hs_code_jp → declaration_elements → units_per_package → ...
→ warehouse → requires_qc → remarks → last_used_date
→ create_by → create_time → is_deleted → update_by → update_time
```

**建议调整顺序**（按业务逻辑分组）:
```
1. 标识: id
2. 货号: master_code → sub_code → jan_code
3. 名称: name_ja → name_en → name_zh
4. 图片: image_url
5. 基础属性: color_name → material → material_ja → origin → category → status
6. 计量: unit → quantities → carton_qty → amount_rmb
7. 单品尺寸: length_cm → width_cm → height_cm → volume_cbm
8. 重量: net_weight_kg → gross_weight_kg
9. 价格: unit_price_rmb → tax_point → tax_rate
10. 报关: hs_code → hs_code_jp → declaration_elements
11. 外箱: units_per_package → package_length_cm → package_width_cm → package_height_cm → package_volume_cbm → package_weight_kg
12. 仓储: warehouse → requires_qc
13. 其他: remarks → last_used_date
14. 审计: create_by → create_time → update_by → update_time → is_deleted
```

> **说明**: DB-11 的 `product` 表字段顺序已较为合理，与 Entity 基本对齐。上述顺序是理想顺序建议，不影响功能，仅提升可读性。

#### P2-2: `procurement` 表字段顺序调整

**现状**（Entity 声明顺序）:
```
id → factory_id → product_code → sub_product_code → material → requires_qc → quantity
→ price_rmb → exchange_rate → tax_point → billing_type → estimated_price_jpy
→ customs_remarks → instruction_manual → order_date → factory_ship_date → planned_ship_date
→ actual_ship_date → product_lead → japan_lead → china_lead → destination → customer_company
→ status → linked_demand_id → linked_demand_item_id → create_by → create_time
→ update_by → update_time → is_deleted
```

**建议调整顺序**:
```
1. 标识: id
2. 关联: factory_id → linked_demand_id → linked_demand_item_id
3. 商品: product_code → sub_product_code → material → requires_qc → quantity
4. 价格: price_rmb → exchange_rate → tax_point → billing_type → estimated_price_jpy
5. 报关: customs_remarks → instruction_manual
6. 日期: order_date → factory_ship_date → planned_ship_date → actual_ship_date
7. 担当: product_lead → japan_lead → china_lead
8. 发货: destination → customer_company
9. 状态: status
10. 审计: create_by → create_time → update_by → update_time → is_deleted
```

> **注意**: `linked_demand_id` 和 `linked_demand_item_id` 在 DB-02 文档中未记录（v1.6.0 新增），需补充文档。

#### P2-3: `logistics_plan` 表字段顺序调整

**建议调整顺序**:
```
1. 标识: id → plan_code
2. 关联: procurement_id → factory_id

4. 尺寸: cargo_length_cm → cargo_width_cm → cargo_height_cm → cargo_weight_kg → cargo_volume_cbm
```
