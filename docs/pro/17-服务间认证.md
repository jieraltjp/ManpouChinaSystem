# 服务间认证

> **视角**：后端开发工程师
> **原则**：JWT Token 在微服务间透明传递，下游服务验证签名
> **状态**：脚手架已就绪，服务间调用已可执行
> **最后更新**：2026-04-20

---

## 1. 当前认证架构

```
外部请求（前端）
    │
    ▼
API Gateway（18080）── JWT 验签（公钥来自 classpath:keys/public.pem）
    │
    ▼
user-service（18081）── JWT 签发（私钥签 RS256）
```

**服务间调用**：
```
procurement-service → warehouse-service
    │
    ▼
请求头：Authorization: Bearer <token>
    │
    ▼
warehouse-service JwtFilter 验签（公钥来自 classpath:keys/public.pem）
```

|------|:-------:|:-------:|------|
| user-service | ✅ RS256 | ✅ | 脚手架阶段即有签发能力 |
| manpou-allinone | ✅ RS256 | ✅ | Phase 0 聚合服务 |
| warehouse-service | ✅ RS256 | ✅ | 脚手架即有签发能力 |
| procurement-service | ✅ RS256 | ✅ | 脚手架即有签发能力 |
| customs-service | ✅ RS256 | ✅ | 脚手架即有签发能力 |
| logistics-service | ✅ RS256 | ✅ | 脚手架即有签发能力 |
| finance-service | ✅ RS256 | ✅ | 脚手架即有签发能力 |
| notification-service | ✅ RS256 | ✅ | 脚手架即有签发能力 |
| product-service | ✅ RS256 | ✅ | 脚手架即有签发能力 |
| api-gateway | ❌ | ✅ RS256 | 仅验签（不持有私钥） |

> **⚠️ 脚手架安全警告**：所有 8 个微服务 + manpou-allinone 均持有**相同的** `private.pem`，且每个服务均有 `JwtService.generateAccessToken()` 方法。这意味着当前架构下，任意一个服务理论上都可以签发任意身份的 JWT Token，属于 Phase 3 之前的安全局限。生产环境必须将签发权收归 user-service（或其他指定的签发中心服务）。

**关键文件**（所有服务一致）：
```
src/main/resources/keys/
├── private.pem   # 所有服务均有相同副本（⚠️ 生产需移除，仅签发中心持有）
└── public.pem    # 全部服务（验签）
```

## 3. 服务间调用示例

### 3.1 同步调用（RestTemplate / WebClient）

```java
// procurement-service 调用 warehouse-service
@RestController
@RequiredArgsConstructor
public class ProcurementController {
    private final RestTemplate restTemplate;

    @PostMapping("/api/v1/purchase-orders/{id}/receive")
    public Result<Void> notifyWarehouse(@PathVariable Long id) {
        String token = obtainServiceToken(); // 从 SecurityContext 获取当前用户 Token
        HttpHeaders headers = new HttpHeaders();
        headers.setBearerAuth(token);  // 将前端 Token 透传给下游
        headers.setContentType(MediaType.APPLICATION_JSON);

        HttpEntity<WarehouseReceiveCmd> req = new HttpEntity<>(cmd, headers);
        restTemplate.postForEntity(
            "http://warehouse-service:18084/api/v1/warehouse/receive",
            req, Void.class);
        return Result.ok();
    }

    private String obtainServiceToken() {
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        if (auth instanceof JwtAuthenticationToken jwt) {
            return jwt.getToken().getTokenValue();
        }
        throw new IllegalStateException("No JWT in SecurityContext");
    }
}
```

## 3. 服务账户（Phase 3 扩展）

当前实现使用前端用户 JWT 透传。
Phase 3 扩展为**服务账户**：

```yaml
# service-account 表（V4 迁移）
service_account:
  id          BIGINT PRIMARY KEY
  service     VARCHAR(64)   -- 服务名：warehouse-service
  kid         VARCHAR(64)   -- 密钥 ID
  scopes      JSON         -- 权限范围：["warehouse:write","warehouse:read"]
  is_active   TINYINT(1)
  create_time DATETIME(3)
```

服务启动时从 Nacos 拉取服务账户 Token，下游调用时使用服务账户 Token（不依赖前端用户 Token）。

### 3.1 Phase 3 扩展方向：签发权集中化

当前脚手架阶段所有服务均可签发 JWT（每个服务持有相同 `private.pem`），这是主要安全风险。Phase 3 需实施以下改造：

- **签发权收归 user-service**：其他服务移除 `private.pem`，仅保留 `public.pem` 用于验签
- **签发中心化**：所有内部服务间调用的 Token 均通过 user-service 签发的服务账户 Token
- **private.pem 安全管控**：`private.pem` 仅部署在 user-service，禁止分发至其他服务
- **密钥轮换**：引入 kid 机制支持多版本密钥并行与滚动切换

```sql
-- Phase 3 签发中心化后的 private.pem 分布
-- user-service:           ✅ private.pem + public.pem（签发中心）
-- manpou-allinone:         ❌ 仅 public.pem（移除 private.pem）
-- 其他 7 微服务:           ❌ 仅 public.pem（移除 private.pem）
-- api-gateway:             ❌ 仅 public.pem（仅验签）
```

## 4. AR-002 风险

> **AR-002**：脚手架阶段所有服务持有签发私钥（🔴 高）

**当前缓解措施**：
- Gateway 拦截外部请求，内部服务通过 K8s NetworkPolicy 隔离
- Phase 3 引入服务账户 + mTLS（双向 TLS）

**Phase 3 根治措施**：
- private.pem 签发权收归 user-service，其他服务仅保留 public.pem
- 所有内部 Token 由 user-service 统一签发服务账户 JWT
