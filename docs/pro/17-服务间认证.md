# 服务间认证

> **视角**：后端开发工程师
> **原则**：JWT Token 在微服务间透明传递，下游服务验证签名
> **状态**：脚手架已就绪，服务间调用已可执行
> **最后更新**：2026-04-20

---

## 1. 当前认证架构

```
外部请求（前端）
    │
    ▼
API Gateway（18080）── JWT 验签（公钥来自 classpath:keys/public.pem）
    │
    ▼
user-service（18081）── JWT 签发（私钥签 RS256）
```

**服务间调用**：
```
procurement-service → warehouse-service
    │
    ▼
请求头：Authorization: Bearer <token>
    │
    ▼
warehouse-service JwtFilter 验签（公钥来自 classpath:keys/public.pem）
```

---

## 2. 现状：各服务均有 JWT 验签能力

| 服务 | JWT 签名 | JWT 验签 | 状态 |
|------|:-------:|:-------:|------|
| user-service | ✅ RS256 | ✅ | 签发方 |
| api-gateway | ❌ | ✅ RS256 | 验签方 |
| 其他 7 服务 | ❌ | ✅ RS256 | 消费者 |

**关键文件**（所有服务一致）：
```
src/main/resources/keys/
├── private.pem   # user-service 独有（签发）
└── public.pem   # 全部服务（验签）
```

## 3. 服务间调用示例

### 3.1 同步调用（RestTemplate / WebClient）

```java
// procurement-service 调用 warehouse-service
@RestController
@RequiredArgsConstructor
public class ProcurementController {
    private final RestTemplate restTemplate;

    @PostMapping("/api/v1/purchase-orders/{id}/receive")
    public Result<Void> notifyWarehouse(@PathVariable Long id) {
        String token = obtainServiceToken(); // 从 SecurityContext 获取当前用户 Token
        HttpHeaders headers = new HttpHeaders();
        headers.setBearerAuth(token);  // 将前端 Token 透传给下游
        headers.setContentType(MediaType.APPLICATION_JSON);

        HttpEntity<WarehouseReceiveCmd> req = new HttpEntity<>(cmd, headers);
        restTemplate.postForEntity(
            "http://warehouse-service:18084/api/v1/warehouse/receive",
            req, Void.class);
        return Result.ok();
    }

    private String obtainServiceToken() {
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        if (auth instanceof JwtAuthenticationToken jwt) {
            return jwt.getToken().getTokenValue();
        }
        throw new IllegalStateException("No JWT in SecurityContext");
    }
}
```

### 3.2 异步调用（Kafka）

```
procurement-service 签发事件
    │
    ▼ Kafka Topic: procurement.events
warehouse/customs/logistics/finance/notification 消费
    │
    ▼ 各服务使用自身 classpath public.pem 验签事件中的 JWT
```

## 3. 服务账户（Phase 3 扩展）

当前实现使用前端用户 JWT 透传。
Phase 3 扩展为**服务账户**：

```yaml
# service-account 表（V4 迁移）
service_account:
  id          BIGINT PRIMARY KEY
  service     VARCHAR(64)   -- 服务名：warehouse-service
  kid         VARCHAR(64)   -- 密钥 ID
  scopes      JSON         -- 权限范围：["warehouse:write","warehouse:read"]
  is_active   TINYINT(1)
  create_time DATETIME(3)
```

服务启动时从 Nacos 拉取服务账户 Token，下游调用时使用服务账户 Token（不依赖前端用户 Token）。

---

## 4. AR-002 风险

> **AR-002**：无服务间认证（🔴 高）

**缓解措施**：
- 所有服务使用相同公钥验签，无法伪造 Token
- Gateway 拦截外部请求，内部服务通过 K8s NetworkPolicy 隔离
- Phase 3 引入服务账户 + mTLS（双向 TLS）
