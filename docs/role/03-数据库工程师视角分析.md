# 角色分析：数据库工程师（Database Engineer）

> **视角**：数据建模 → 迁移管理 → 性能优化 → 数据安全
> **核心问题**：数据模型能否准确表达业务语义？迁移是否安全可回滚？

---

## 1. 当前数据库架构

### 1.1 开发环境

| 配置 | 值 |
|------|-----|
| 类型 | H2 内存数据库（开发）/ MySQL 8（生产） |
| 引擎 | InnoDB |
| 字符集 | utf8mb4 / utf8mb4_unicode_ci |
| JPA ddl-auto | update（开发）/ validate（生产） |
| Flyway | 禁用（开发）/ 启用（生产） |

### 1.2 审计字段规范（强制）

所有业务表必须包含：

```sql
create_time    DATETIME(3)  NOT NULL  DEFAULT CURRENT_TIMESTAMP(3)
update_time    DATETIME(3)  NOT NULL  DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3)
create_by      VARCHAR(64)  NOT NULL  DEFAULT ''
update_by      VARCHAR(64)  NOT NULL  DEFAULT ''
is_deleted     TINYINT     NOT NULL  DEFAULT 0
```

    orderer_id      BIGINT          NOT NULL,

    contact_phone   VARCHAR(20),
    shipping_address VARCHAR(256),
    is_export       TINYINT         NOT NULL  DEFAULT 0,
    status          VARCHAR(20)     NOT NULL,              -- DRAFT/PENDING/APPROVED...
    factory_code    VARCHAR(32),

## 5. 索引设计规范

### 5.1 必加索引

| 场景 | 索引类型 | 说明 |
|------|---------|------|
| 主键 | PRIMARY | 自动 |
| 唯一字段 | UNIQUE | order_no, arrival_no, document_no |
| 外键 | INDEX | order_id, purchase_order_id |
| 状态查询 | INDEX | status |
| 审计查询 | INDEX | create_by, create_time |
| 软删除 | INDEX | is_deleted |

### 5.2 禁止

- ❌ 不加索引直接上线
- ❌ 全表扫描用于业务查询
- ❌ 在大表上建过多索引（影响写入性能）
